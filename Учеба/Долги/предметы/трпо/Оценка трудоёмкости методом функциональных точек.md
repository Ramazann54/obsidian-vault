# Метод функциональных точек (Function Point Analysis, FPA)

## Определение
**Метод функциональных точек** — это способ **оценки размера программного продукта с точки зрения пользователя**, то есть по объему полезной функциональности, которую система предоставляет, а не по числу строк кода. Метод был разработан для того, чтобы максимально уйти от деталей реализации и оценивать ПО через наблюдаемую пользователем функциональность. :contentReference[oaicite:0]{index=0}

---

## Зачем нужен метод
Метод функциональных точек применяют, когда нужно:

- оценить размер разрабатываемой системы;
- сравнивать проекты независимо от языка программирования и платформы;
- получить основу для дальнейшей оценки трудоёмкости и сроков;
- оценивать как новую разработку, так и доработку существующего продукта. :contentReference[oaicite:1]{index=1}

---

## Основная идея метода
Вместо оценки по коду метод рассматривает:

- **данные**, которыми оперирует система;
- **транзакции**, через которые пользователь взаимодействует с системой.

То есть оценивается не внутренняя реализация, а **функциональность на границе системы**. :contentReference[oaicite:2]{index=2}

---

## Преимущества метода
- не зависит от языка программирования;
- не зависит от платформы реализации;
- подходит уже на ранних стадиях проекта;
- обеспечивает единообразный подход к оценке разных проектов. :contentReference[oaicite:3]{index=3}

## Недостатки метода
- сам по себе **не даёт трудоёмкость напрямую**;
- в оценке есть доля субъективности;
- используются **косвенные**, а не прямые измерения. :contentReference[oaicite:4]{index=4}

---

## Типы оценки
Метод предусматривает **3 типа оценки**:

### 1. Проект разработки
Оценивается функциональность, поставляемая пользователю в первом релизе.

### 2. Проект развития
Оцениваются изменения существующей системы:
- добавление функций;
- изменение функций;
- удаление функций.

### 3. Продукт
Оценивается уже существующий установленный продукт. :contentReference[oaicite:5]{index=5}

---

## Шаги метода
Последовательность анализа:

1. определить тип оценки;
2. определить область оценки и границы продукта;
3. подсчитать функциональные точки, связанные с данными;
4. подсчитать функциональные точки, связанные с транзакциями;
5. определить суммарное количество **невыровненных функциональных точек** (**UFP**);
6. определить **фактор выравнивания** (**VAF**);
7. рассчитать количество **выровненных функциональных точек**. :contentReference[oaicite:6]{index=6}

---

## Граница системы
При анализе важно определить, где проходит **граница приложения**:

- что считается внешним по отношению к продукту;
- какие транзакции пересекают границу системы;
- какие данные поддерживаются самим приложением, а какие являются внешними. :contentReference[oaicite:7]{index=7}

---

## Основные элементы метода

## 1. Компоненты данных

### ILF — Internal Logical File
**Внутренний логический файл** — логически связанная группа данных, которая:
- находится **внутри** границ приложения;
- поддерживается этим приложением.

Примеры:
- таблица БД;
- логический объект «Клиент»;
- файл данных, поддерживаемый системой.

### EIF — External Interface File
**Внешний интерфейсный файл** — логически связанная группа данных, которая:
- находится **вне** границ приложения;
- используется системой;
- но поддерживается другим приложением. :contentReference[oaicite:8]{index=8}

---

## 2. Компоненты транзакций

### EI — External Input
**Внешний ввод** — транзакция, при которой данные поступают в систему извне и могут изменять ILF.

Примеры:
- ввод данных в форму;
- создание записи;
- изменение объекта.

### EO — External Output
**Внешний вывод** — транзакция, при которой данные выходят из системы и при этом выполняется обработка или вычисление.

Примеры:
- отчет;
- сформированная ведомость;
- сообщение, полученное после обработки данных.

### EQ — External Inquiry
**Внешний запрос** — транзакция, при которой есть и ввод, и вывод, но на выходе возвращаются «сырые» данные без сложной обработки.

Пример:
- поиск по форме и вывод найденных записей. :contentReference[oaicite:9]{index=9}

---

## Параметры сложности

## Для данных
Используются:

### DET (Data Element Type)
Уникальное неповторяемое поле данных.

Пример:
- имя клиента — 1 DET;
- адрес, если он состоит из 9 полей, — 9 DET.

### RET (Record Element Type)
Логическая группа данных.

Пример:
- адрес;
- паспорт;
- номер телефона. :contentReference[oaicite:10]{index=10}

---

## Для транзакций
Используются:

### DET
Уникальные поля данных, участвующие в транзакции.

### FTR (File Type Referenced)
Количество различных файлов типа ILF/EIF, которые транзакция читает или изменяет. :contentReference[oaicite:11]{index=11}

---

## Подсчёт UFP для данных
Сначала определяется сложность ILF и EIF по матрице, зависящей от:

- количества **DET**;
- количества **RET**.

После этого назначается вес:

| Сложность | ILF | EIF |
|---|---:|---:|
| Low | 7 | 5 |
| Average | 10 | 7 |
| High | 15 | 10 |

То есть каждый информационный объект получает определённое количество **невыровненных функциональных точек**. :contentReference[oaicite:12]{index=12}

---

## Подсчёт UFP для транзакций
Для EI, EO и EQ сложность определяется по матрицам на основе:

- числа **DET**;
- числа **FTR**.

После определения сложности используется вес:

| Сложность | EI / EQ | EO |
|---|---:|---:|
| Low | 3 | 4 |
| Average | 4 | 5 |
| High | 6 | 7 |

Таким образом оценивается каждая транзакция системы. :contentReference[oaicite:13]{index=13}

---

## Формула суммарного UFP
Общий объём реализации в невыровненных функциональных точках:

**UFP = сумма оценок всех ILF + EIF + EI + EO + EQ**. :contentReference[oaicite:14]{index=14}

---

## Фактор выравнивания VAF
После подсчёта UFP учитываются **общесистемные характеристики**, которые усложняют разработку.

Используется **14 параметров влияния**, например:

- обмен данными;
- распределённая обработка;
- производительность;
- ограничения по аппаратным ресурсам;
- транзакционная нагрузка;
- интенсивность взаимодействия с пользователем;
- эргономика;
- интенсивность изменения данных;
- сложность обработки;
- повторное использование;
- удобство инсталляции;
- удобство администрирования;
- портируемость;
- гибкость. :contentReference[oaicite:15]{index=15}

Каждый параметр оценивается по шкале от **0 до 5**:
- 0 — нет влияния;
- 1 — случайное;
- 2 — умеренное;
- 3 — среднее;
- 4 — значительное;
- 5 — существенное. :contentReference[oaicite:16]{index=16}

### Суммарный эффект
**TDI = Σ DI**

### Формула VAF
**VAF = 0.65 + 0.01 × TDI**

Фактор выравнивания может изменить оценку примерно на **±35%**. :contentReference[oaicite:17]{index=17}

---

## Итоговые формулы

## 1. AFP — Adjusted Function Points
Для обычного приложения:

**AFP = UFP × VAF**

Это количество **выровненных функциональных точек**. :contentReference[oaicite:18]{index=18}

## 2. DFP — Development Function Point
Для проекта разработки:

**DFP = (UFP + CFP) × VAF**

где **CFP** — дополнительные функциональные точки, например на миграцию данных. :contentReference[oaicite:19]{index=19}

## 3. EFP — Enhancement Function Point
Для проекта доработки:

**EFP = (ADD + CHGA + CFP) × VAFA + DEL × VAFB**

где:
- **ADD** — добавленная функциональность;
- **CHGA** — изменённые функции после модификации;
- **CFP** — дополнительная функциональность;
- **VAFA** — фактор выравнивания после проекта;
- **DEL** — удалённая функциональность;
- **VAFB** — фактор выравнивания до начала проекта. :contentReference[oaicite:20]{index=20}

---

## Как связать FPA с трудоёмкостью
Сам метод функциональных точек **не даёт напрямую трудоёмкость**. Он оценивает **размер** продукта. Далее:

- если у компании есть своя статистика, можно перевести FP в человеко-часы;
- если статистики нет, можно использовать **COCOMO II**;
- в лекции предлагается как стартовая оценка: **20 чел.-часов на 1 UFP**, с возможной корректировкой по квалификации команды. :contentReference[oaicite:21]{index=21}

---

## Краткий алгоритм ответа на экзамене
Если нужно быстро рассказать тему, можно говорить так:

> Метод функциональных точек — это способ оценки размера ПО с точки зрения пользователя. В нём оцениваются данные и транзакции системы: ILF, EIF, EI, EO, EQ. Сначала считают невыровненные функциональные точки UFP, затем учитывают 14 системных факторов через VAF, после чего получают выровненную оценку AFP. Метод не зависит от языка программирования и платформы, удобен на ранних этапах, но сам по себе не даёт трудоёмкость напрямую — для этого нужны статистика компании или метод COCOMO II. :contentReference[oaicite:22]{index=22}

---

## Плюсы и минусы — кратко

### Плюсы
- независимость от языка и платформы;
- можно применять рано;
- удобно сравнивать проекты;
- оценка ближе к пользовательской функциональности.

### Минусы
- есть субъективность;
- нужен опыт правильного выделения компонентов;
- трудоёмкость получается только через дополнительные модели. :contentReference[oaicite:23]{index=23}
