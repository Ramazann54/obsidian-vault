# Модель быстрой разработки приложений (RAD)

**Цель:** быстрый выпуск первых версий ПО с активным участием заказчика.  
**Особенности:** инкрементная стратегия, работа малыми группами, использование готовых компонентов.

## Основные принципы

- Разработка малыми командами (3–7 чел.) → лучше управляемость.
    
- Использование готовых компонентов → быстрое получение прототипа.
    
- Чёткий цикл ≤ 3 месяцев → высокая эффективность.
    
- Ориентирована на информационные системы.
    

## Этапы RAD

### 1. Бизнес-моделирование

- Определяется **информационный поток** между бизнес-функциями.
    
- Вопросы:
    
    - Какая информация руководит процессом?
        
    - Какая генерируется? Кто генерирует?
        
    - Где применяется и кто обрабатывает?
        

### 2. Моделирование данных

- Информационный поток → **объекты данных**.
    
- Определяются атрибуты и связи объектов.
    

### 3. Моделирование обработки

- Определяются **преобразования объектов данных** для бизнес-функций.
    
- Создаются описания обработки (добавление, изменение, удаление, исправление).
    

### 4. Генерация приложения

- Используются **ЯП 4-го поколения** и повторно используемые компоненты.
    
- Автоматизация конструирования через утилиты.
    

### 5. Тестирование и объединение

- Повторно используемые компоненты уже протестированы → меньше времени на тест.
    
- Новые элементы тестируются отдельно.
    

## Ограничения RAD

- Требует много людей для больших проектов.
    
- Применима только для модульных приложений, где производительность не критична.
    
- Не подходит при высоких технических рисках (новые технологии).
    
- Оптимальна для небольших, стандартных проектов с участием заказчика.
